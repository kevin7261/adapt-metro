# RWD Maps 版面路網畫線 (route-rwd-draw)

把 [[route-hillclimb]] 的**縮減網格**佈局重繪成示意圖折線：方向數可選
**任意／4／8／16**（見下方「線方向數」）。此 skill 是這個畫線規則的**唯一權威依據**。

> **硬規則：八方向約束以「版面 pixel」為準**——H/V/45° 是在 SVG 畫面座標上判定，
> **不是整數格索引**。預設圖**隨板面變形**（欄寬 ≠ 列高是常態）；工具列「版面」下拉
> 可改成固定尺寸（網頁／手機／IG，類似 Illustrator artboard）——以該寬高當畫線座標
> 系，SVG `viewBox` letterbox 置中，用來模擬 RWD。格座標的對角線在畫面上
> 不是 45°，所以折線在**當下畫布的像素空間**重建；視窗／版面尺寸一變就在新像素座標
> **重跑 buildRwdMap**（同尺寸的 render 用快取）。曾試過「固定正方虛擬格＋等比
> letterbox」讓 layout 凍結（與裝置預設無關），**使用者裁決不要**——隨板面變形才是對的；
> 裝置預設是後來另加的模擬開關。
> （附帶效果：非正方格時 A* 只能走 H/V 步——斜步在像素上不是 45°；lattice 恰為
> 正方時才自動開 8 方向。）
> 實作：`src/stores/rwdMap.js` 的 `buildRwdMap(segs, pos, opts)`；UI 在 `D3Tab.vue`
> （`layer.type === 'rwd'`），layer group「RWD Maps」；版面預設見 `src/lib/rwdFrames.js`。

## 資料流

```
Hill Climbing 圖層的 縮減網格（compactGrid cells）
  →（形狀圖層＋成方）循環可動方形頂點但不破方；LLM 對齊／評價釘回成方座標
  → 換成像素座標（等寬格心）
  → mergeParallelSegs：共端點的平行邊收成一條交錯線（見下）
  → buildRwdMap：每段選 H/V/45 折線；**成方邊（兩端為 frozen members＝灰白 highlight）強制 S→T、絕對不可改彎**
  → 黑點沿選定折線弧長平均放回
```

形狀圖層（`orig-shape`／`rot-shape`）有成方時，灰白 highlight 的成方邊在 RWD
**絕對不可改變**（`opts.frozenIds`＋`shapeLock`；不要求像素 H/V；成對 rip
也只能讓對方繞過，不可拆成方邊；見 [[route-shape-align]]）。

> **不變式：所有彩色點（cut 端點）必定落在網格交叉點上。** 它們是整數格佈局的格心，
> RWD 只在其間畫折線、不得移動彩色點（黑色內部站才沿折線弧長放回）。曾試過把 degree-2
> 同線轉折點折進 interior 以拉直階梯（`mergeThroughSegs`），會把彩色點移出格交叉點——
> **使用者裁決不要、已移除**。要減少階梯得從版面（hillclimb/縮減網格）讓那些節點更共線，
> 不能在 RWD 這端搬點。

**同軌路線合併（`mergeParallelSegs`，使用者裁決：RWD 要跟匯入的 network 一樣，
重疊就一條、不錯開成多條平行線）**。判準一律看**匯入的真實幾何**，不管官方圖語意。
兩個機制：

- **(A) 同端點**：兩條以上骨架邊共用**同兩個 cut 端點**（快車直達＋普通車，或任何
  共端點路線）→ 站數最多的當代表、其餘 routes／routeColors 併入、其 seg 丟棄。
- **(B) 幾何重疊（快車跨過中間轉乘節點）**：快車邊 E 的**整條 `edge.geom`** 都貼在
  另一組邊的軌道上（`maxDev < TAU≈10m`），即使它跳過中間 junction、端點對不上——
  因為快車物理上就走同一條軌（同軌路線在資料裡共用同一組座標、偏差≈0），network
  本來就畫在那條軌上。找出所有落在 E 軌道上的邊、若它們連通 E 的兩端且總站數 > E
  站數（E 是較稀疏的快車側），就把 E 併入這條鏈的每一段、E 的 seg 丟棄。TAU 很緊
  （~10m），相隔數十公尺以上的**平行但不同軌**不會被誤併。

**涵蓋性防呆（關鍵，絕不可省）**：只有當被併掉的邊**每一站都已在保留路徑上**才併
——否則被丟掉的 seg 上那些**獨有站會孤立**（畫面上有點卻沒有線，重大錯誤）。幾何重疊
**不等於**同一條線：一條獨立線（如 Sydney Metro）可以和市郊鐵路**共軌**但有自己的
車站，這種**一律不併、維持分開**。安全可併的例子＝2 站直達快車（兩站都是走廊上的
共用節點、無獨有站）。機制 (A)(B) 都套此防呆。實測全 224 城 **0 孤立站**。

代表邊 cls 若因此帶 ≥2 色 → coline（strokesOf 交錯畫）。環線（`e.a===e.b`）不參與。
純函式、依序決定（router 對 seg 順序敏感），routing 前套用。附帶效果：線變少 →
衝突也變少。全 224 城 0 fallback、0 例外、0 孤立站。

**環邊正反去重（`buildRwdMap` 內，routing 前）**：`buildHcGraph` 會把環（含
單線自交叉，如拿坡里 Line 1 的 Vomero 髮夾彎——`skeleton.js` 的自交叉補黃點使該線
被切成 `cls==='loop'` 的環）以**兩個方向**輸出——每條環邊出現兩次、端點對調。兩份畫在
同一條軌上，router 為避重疊會把它們平行錯開成**糾纏的雙層輪廓**。故 routing 前把
`cls==='loop'` 且無序端點對 `{a,b}` 已出現過的 seg 丟掉，每條環邊只畫一次：孿生邊是
冗餘幾何（同 routes／色），丟掉無損，環改以單一乾淨環呈現。實測封閉性不變（環城市
去重後所有環節點仍偶數度＝封閉）：Glasgow 4、Gurgaon 8→4、Singapore 28→24、
拿坡里（自交叉假環）6→3（乾淨直角三角形）。真環線（單向遍歷、各邊端點對唯一）不受影響。

- 新圖層由 Layers 面板 RWD Maps group 的 **+** 建立：選一個 Hill Climbing 圖層
  （存在 `layer.sourceLayerId`）＋**循環的 9 個結果之一**（論文①〜⑧＋LLM 對齊的循環，即 stroke/rect/milp/force/lsq/octi/path/sat/llm 九條鏈的
  端點移動+直線縮減+中位集中+縮減網格循環，存在 `layer.compact`）——來源即該鏈的
  循環結果；舊圖層的 `'hc'`（基本循環）僅作 fallback、對話框不再提供。
- Tab 有 3 個視圖：**循環結果**（輸入，id 沿用 'hc-compact'）、**RWD 路網**（結果）與 **LLM調整**
  （同一套 RWD 畫線，欄寬列高改由 LLM 推理的區間權重決定，見 [[route-llm-grid]]
  與下方「LLM 調整」節）；工具列顯示段數與轉折直方圖（直線/單折/雙折/兜底/強制重疊）。
- **8 視圖畫廊**（RWD Maps group 的方格鈕 → `openRwdGalleryTab`）：全城市預算縮圖，
  每城 9 種縮減網格（基本 hc＋論文①〜⑧）× (縮減網格 | RWD 路網)＝18 格。
  離線由 `computeCityRwdViews`（viewGeometry.js）算、存 `data/metro/rwd-maps/<id>.json`，
  `RwdGallery.vue`＋`RwdViewGrid.vue` 載入；點一格＝建立 metro→d3→HC(原始)→RWD(該縮減)。

## 候選折線（轉折數由少到多）

方向判準（像素座標，容差 ~0.75px）：`Δy≈0`→H、`Δx≈0`→V、`|Δx|≈|Δy|`→45°、其餘不合法（X）。

**線方向數（`opts.dirs` ∈ {0, 4, 8, 16}，工具列四選一、預設 8）**：允許用到哪一級的方向。
**方向級優先＝tie-break，不是硬閘（使用者最終裁決）**：能用 0/90 就不用 45、能用 45
就不用 22.5°/67.5°、能用離散族就不用任意角——但這是**同衝突數時**的優先；「明明有
22.5 的畫法不重疊卻沒有」是錯的。衝突數（含貼線重疊）排在方向級之前：零衝突的 22.5°
贏過有重疊的 45°。
- **任意方向**（`dirs === 0`）：任何 S→T 直線都合法（**直線優先**，Rule 0 走廊含任意角）；
  非 H/V 的角度不限於 45/22.5。繞行時仍可展開完整 16 方向候選族，並另產「同角平行偏移」
  （被擋時先貼旁走、角度不變）。衝突／貼線把任意角腿當 `G` 族（`ang`＋法向 key）；
  角度差 ≤≈3.4°（`ANG_PARALLEL`）才算同族——**非不得已很靠近 → 斜邊同角**。收尾
  `angleBalancePass`：節點出射角盡量均分；過近的可動折線對齊到鄰邊同角。直線不為均分增折。
  `pickBest` 方向級：H/V → 45 → 22.5 → 任意角。
- **4 方向**（只 H/V）：不產任何 45° 候選；45° 直線視為非法（走 L）；`legalDetours` 的
  H/V 平行偏移改**純直角**（V–H–V／H–V–H，非 45° 三角）；A* 的正方格 8 方向與 45°
  macro 斜出**都關閉**（純直角佈線）；pass 2 的 **L→45（`l45Pass`）跳過**。實測全城 45°＝0。
  **判斷方式與 8/16 方向完全一致（使用者規則：三個方向級同一套判斷）**：對角段用
  **同一組骨架**（L、雙折 50/25/75、四折／多折階梯），唯一差別是每條 45° 腿展開成
  H/V（兩種順序都試、合併同向腿、依折數重排）；A* 救援照用（純直角步進）。
  ⚠️ 45° 腿展開（整條共用一個 vhFirst）產不出「真 Z 字」——V–H–V 的中段要 H、兩端
  同向 V，展開只會生出多一折的階梯。所以 4 方向**另外直接生成真 Z 字**：
  `V–(tΔy)–H–V` 與 `H–(tΔx)–V–H`、t ∈ {50%, 25%, 75%}（`pushCand`，zRank 同 50→25→75）。
  沒有這組，「可以 25% 轉折卻畫成階梯壓線」的錯就會出現。
  分點順序 **50% → 25% → 75%**：衝突數與重疊長度都相同時 50% 一定優先；為了**不要
  重疊太多**，25%/75% 的重疊較短時可蓋過 50%（見「候選選擇序」④→⑤）；畫完仍有
  重疊的段另有 `overlapReducePass` 收尾再試同折數分點（見下）。端點、連接關係與
  局部走向不變（拓撲不變）。8/16 方向不產直角樓梯（對角改用 45° 候選）。收尾的順接
  軟調整（softPass）與降折重掃在 4 方向照常跑。
- **8 方向**（＋45°）：現狀（H/V/45°）。
- **16 方向**（＋22.5°/67.5°）：加 22.5° 族（E±=22.5°、F±=67.5°）。幾何族：`dirOf`／
  `lineKey`（E±:y∓T22·x、F±:x∓T22·y，`T22=tan22.5°`）／`spanOf`（F± 用 y）／`perpGap`
  （E/F 除 `NORM_EF`）全支援；`legsCross` 是一般線段相交、天生支援。**22.5° 候選（`skew:true`）
  在 `pickBest` 同場競爭**（曾經是硬閘「45 全違法才放行」——使用者裁決改掉：明明有
  22.5 的畫法不重疊就要用）：衝突數第一、近距貼線第二、skew 第三 → 零衝突的 22.5°
  贏過重疊**或近距貼線**的 45°；兩者都乾淨時仍 45° 優先，所以不會無故畫 22.5°。
  **候選家族（2026-07-20 擴充）**：單折（斜＋軸向，4 種）之外另有**雙折 22.5 家族**
  （斜–軸–斜／軸–斜–軸，分點 50/25/75）與 **45＋22.5 混合家族**（45–軸–斜／斜–軸–45，
  使用者規則：**同一條路可以 45 與 22.5 組合**）——所有斜腿仍以軸向腿隔開（斜轉斜
  禁止不變）、全部標 `skew:true`（同衝突數時 45 級照樣優先）；每條腿須長過 2×TOL
  （次像素腿會被 dirOf 誤判）。沒有這些家族時，「45 全衝突、單折 22.5 也衝突、但
  45＋22.5 混合可以不重疊」的段會被迫貼線／繞遠／forced。
  **「能 45 就不用 22.5」的兩個補強（2026-07-20，台北藍線案）**：
  - **joint 重算也守方向級**：`jointReroutePairs` 的候選依 `(skew, bends)` 排序、
    組合目標＝(skew 腿數, 總折數) 字典序——曾只比總折數，22.5 單折會贏過乾淨的
    45 雙折。
  - **22.5 → 45 升級（`deskewPass`，收尾每輪，排在 destairPass 之後）**：22.5 是
    「當下 45 全衝突」的暫時解，衝突消解後走廊常已空出，但降折/軟調整都不增折換
    方向級、22.5 一畫就永遠留著。收尾重試：含 22.5/67.5 腿的線只要存在**乾淨的
    45 級候選**（零衝突、近距貼線與硬重疊不得變長、環狀順序不變）就換成折數最少
    的那個——**折數可以增加**（方向級優先於折數）。貼線護欄保住既有裁決：45 形
    要貼著別條線跑時仍保留不貼線的 22.5。
  `rerouteAround` 乾淨候選也依「近距貼線 → skew」取捨；`overlapReducePass` 允許換到
  22.5 級但要求硬重疊或近距貼線嚴格更短（硬重疊不得變長）。45° 雙折的分點優先序固定為
  **50% → 25% → 75%**。**降折（`bendReductionToFixpoint`）與軟調整（`softPass`）仍不得
  把 45 級「降折」到 22.5 級**（那兩個 pass 不是為了消重疊）。
UI：工具列（`StyleBar` RWD 排）的「4方向/8方向/16方向」group button → D3Tab `rwdDirs`
→ `buildRwdMap` opts.dirs（併進 RWD 快取鍵）。

1. **直線** `S→T`：本身已是 H/V/45 → 轉折 0。
2. **單轉折**：45° 先走到與較短邊等長處再接 H/V；純 L 形（先 H 後 V／先 V 後 H）；
   鏡像（H/V 先、45° 收尾）。
3. **雙轉折**（兩種鏡像家族，同折數，各三版 1/2·1/4·3/4）：
   - **(a) 45–H–45 / 45–V–45**（斜–平–斜，**兩端 45°**）：中段落在 1/2，衝突改 1/4、3/4。
   - **(b) H–45–H / V–45–V**（平–斜–平，**兩端軸向、中間 45°**，使用者規則：`hv-45-hv`
     也要有）：45° 走 m 對角、兩端軸向分掉余剩 |Δx−Δy|。兩端與鄰段直線續接時比 (a)
     更順（讀成一條線）。pass 2 依續接角在 (a)(b) 間擇優。
3½. **三轉折混合（使用者案例 `vh-45-vh-45`）**：`axis–45–axis–45` 與 `45–axis–45–axis`
   （axis＝較長軸的 H 或 V）。雙折家族的軸腿只能貼在**起訖兩排**；這一族把中間軸腿
   抬到**中間列/欄**——45 riser 分點 rf 與軸腿分點 hf 各取 {50%, 25%, 75%}（3×3×2
   ＝18 個候選）。起訖兩排走廊都被別的線佔用時，這是唯一的三折解；沒有這族，段會
   被迫壓線（amber）或跳到四折階梯。
4. **交替階梯一般化（使用者規則：`vh-45-vh-45` 或 `45-vh-45` 重覆幾次**不設限**，
   只要能正確畫、折數最少的合法候選勝出；唯一硬規則是「45° 一律接 H/V」＝45° 不相鄰）**：
   45° 腿與軸腿（較長軸的 H 或 V）**任意次交替**，四種頭尾組合都生成——`45…45`
   （斜多一條）、`axis…axis`（軸多一條）、`45…axis` 與 `axis…45`（等數、兩種順序）。
   均勻分點（≤4 折家族另帶 50/25/75 分點變體）；腿短於一格（u）的組合跳過＝段數由幾何
   自然封頂（另加硬頂折數 ≤20 防候選爆炸）。折數不用預先偏好——`pickBest` 的
   「衝突最少 → 折數最少」自然收斂到最短的合法交替。舊有兩種固定家族仍在：
   - **同軸**：45–H–45–H–…–45（ax>ay）或 45–V–45–V–…–45（ay>ax）。每段 riser=m/k、
     余剩全在較大軸拆成 k−1 段。只有 45° 與一個軸向，讀起來較乾淨 → **先試**（k=2 另有
     tread 落點 1/2·1/4·3/4 變體）。
   - **混合（k=3）**：45–H–45–V–45 與 45–V–45–H–45。對角每段取 <m/3（3g<ax 且 3g<ay），
     使 H、V 皆有剩餘各成一段 → 需要同時往兩軸繞開障礙時用。更複雜的排列由 A* 佈線器
     （同樣守 45°→H/V 規則）處理。
   **45° 脚一律被軸向直線隔開，不構成禁止的 45°→45° 角**。採用時計入 `stats.multi`
   （工具列「多折」）。折數絕對優先照舊——多段只在更少段的候選全衝突時才選。
5. **兜底**：以上全衝突才畫原方向直線（非 H/V/45），計入 `stats.fallback`。

**直線鐵律 + 重疊優先於折數（使用者規則）**：可直線就直線（Rule 0，鎖定）。非直線段
則**重疊線最少優先於折數**——L 雖少一折，若共線較長，改採 50／25／75 的 Z 字（或再多
一階梯）。選擇序見下「候選選擇序」。衝突消解後還有**降折重掃**（`stats.straightened`）：
成真直線仍優先；其餘降折不得讓軟共線變長。含**協商式拉直**——更直的畫法只被 1–2 條
線擋住時，拆掉那些線、採用直畫法、擋路線改繞，全部乾淨才提交。唯「頭尾共點雙胞胎」
不可能兩條都直（會完全重疊），必留一條繞行。
> **鐵律「只要可以直線就一定是直線」的預算放寬（`bendReductionToFixpoint`）**：協商式
> 拉直平時要求**總折數嚴格下降**才提交；但當這條因此變成**直線**（`c.bends===0`）且**沒有
> 犧牲任何原本就是直線的擋路線**（`ripsStraight` 檢查 `saves[k+1].bends===0`）時，**不設
> 折數預算**——已經彎的鄰線要多彎幾折都得讓，好成全這條的直線。收斂保證：直線數嚴格
> 遞增、直線絕不被 un-straighten。**但這只在「全部乾淨（無新交叉/重疊）」時提交**——
> 消交叉的重掃/rip-up 仍可拆任何線（含直線），因為殘留交叉（forced）比彎線更糟。

**45°轉45° 禁止（使用者規則）**：折線不得有兩條相鄰腿皆為斜線（不同族）——
H/V 線因此**沒有合法單折繞行**（唯一的單折是 45-45 三角，被禁），只剩雙折平行偏移；
A* 也禁止斜向轉斜向的步進與 macro 接合。

**直角樓梯禁止、對角走向用 45°（使用者規則，最高目標之一）**：折線不得有**連續的
直角轉折**（`H-V-H` 或 `V-H-V` ＝一階一階的樓梯，`vh-vh-vh`）——對角走向的段要用
**45° 斜線**畫，不要畫成直角樓梯。合法：45° 斜線、`45-H-45`、`vh-45-vh`（直角被
45° 隔開，單一 L 也可）、多段 45° 階梯（`45-H-45-H-45`）。不合法：兩個以上**同型
直角轉折相連**（沒有 45° 或直線介入）。45° 與軸向的分界不限 1/2，也可 1/4、3/4
（見雙轉折候選 `[0.5, 0.25, 0.75]`）。
> **A\* 樓梯 → 45°（`destairPass`，2026-07-20 實作）**：非正方版面的 A\* 佈線器只能
> 走 H/V 步（斜步在 pixel 上不是 45°，見下方硬規則），救援出來的路徑是直角樓梯。
> 收尾迭代的**第一步**做後處理：把每條 A\* 畫的線（routed，跳過 squeezed/colinear/
> forced）裡「連續 H/V 交替、≥2 個直角轉折」的樓梯子段抓出來，用**同一套 8/16 方向
> 候選**（含 pixel 空間 45° 腿，不含 skew 級）在子段兩端點之間重畫再接回——折數不得
> 增加、必須含斜腿或嚴格少折、整條線零衝突、貼線不得變長、環狀順序不變、拼接處不得
> 斜轉斜，全部乾淨才換（計入 `stats.diag45`）。殘留樓梯＝候選全放不進的密網角落。

**L→45（使用者最終規則，`stats.diag45`）**：純「水平＋垂直」直角折線，若把其中一邊
改成 45° 不與任何線重疊，就改成 45°＋軸向；**優先選「同一路線在該節點的鄰段也是
同方向 45°」的那一邊**——斜線穿過節點同方向連起來，讀起來是一條線。

**合法直線也要有替代**（頭尾共點的平行邊、走廊已被先前折線佔用），同樣依折數排序：
先試**單折大繞行**（H/V 線的 45° 三角頂點、45° 線的 L 轉角——位移再大也優先於雙折），
再試**雙折平行偏移**（45° 出去、以 d=1→2 個格單位並行、45° 回來）。

## 絕不交叉的多層消解（依序升級）

「路線交叉絕不允許」是最高優先硬規則，候選折線都衝突時依序升級：

1. **衝突消減重掃**：全部 routed 完後重訪每條仍有衝突的線（交叉的兩造都重試），
   以完整資訊重選最少衝突候選，至多 4 輪。**個別重掃把另一條當固定**（`placedOf(li)`
   只排除自己）——互相擋、各自繞不開時解不開。
1b. **成對線一起重算（`jointReroutePairs`，使用者規則：畫了線會交錯就要 2 條線重新
   計算；**出現交叉就要確認兩條線有沒有不交叉的畫法**）**：仍**彼此交錯**（不同族腿
   `legsCross`，非共線貼線）的一對，把兩條**同時拆掉**、其餘所有線固定，聯合搜兩者的
   bounded-bend 候選**組合**（每條前 16 個、皆 bend-ordered），取「兩條都乾淨（對其他
   線＋**彼此不交叉**）且**總轉折數最少**」的一組（折數絕對優先照舊）。跑在**個別重掃
   A\*/rip-up 升級之前**（乾淨組合勝過把單條 A\* 繞成 squiggle）＋重掃之後各一次，迭代
   到不動點（上限 3 輪）。動畫幀（`opts.fast`）跳過。解不掉的才續往下。
   - **入場＝直接偵測真交叉（`crossingPairs`，2026-07-20）**：bbox 先篩、再 `legsCross`，
     **與 forced 旗標完全脫鉤**——曾要求「兩條都 forced」才入場，漏掉最常見的「先畫時
     乾淨、被後畫的 forced 線壓過」交叉對（先畫那條的旗標從未被補上）。
   - **成對 rip 升級（`jointRipPair`，重掃後的 deep 輪）**：bounded-bend 組合放不下時，
     拆掉對方 → A\* 畫自己 → 對方 `rerouteAround` 繞回（兩種順序都試），**全部乾淨才
     提交**——「這對線到底有沒有不交叉畫法」的實質驗證。交叉是最高級錯誤：對方是
     **鎖定直線也解鎖讓路**（殘留交叉比彎線更糟；直線只被動繞、不主動 A\*，繞完仍是
     乾淨直線才保留鎖）。實測全 234 城交叉對 139→112、forced 208→132、折數僅 +0.1%。
2. **A\* 格點佈線器**（`routeLattice`）：在半格 lattice 上找一條**只穿過自由空間**
   的路（由構造保證不交叉、不貼線、不壓點）。正方格時斜向步進是合法 45° → 8 方向；
   **非正方版面只能走 H/V 步**（斜步在 pixel 上不是 45°）＝會產生直角樓梯——收尾的
   `destairPass`（見上「A\* 樓梯 → 45°」）再把樓梯子段換回 45° 折線。端點被同線腿
   佔滿時用 **45° macro 出入口**（斜出＋
   直行一折）接上 lattice；先跑**定向 flood 預檢**（T→S 最佳優先）——封在小口袋秒退，
   並回報「牆」是哪幾條線。轉折數可超過 2（計入多折/繞行）。
   **大環繞禁止（使用者規則 2026-07：只能在最短路線附近尋找可能性，絕不可繞出
   一圈遠路）**：A\*、flood 預檢與 macro 出入口都限制在 **S–T 走廊窗**（兩端點
   bbox 外擴 `LOCAL_PAD = 3×unit`）內，且路徑總成本（長度＋轉折罰）≤
   `3×曼哈頓距 + 8×unit`。走廊內找不到＝視為無路——flood 撞到的走廊牆線回報給
   rip-up 拆牆局部改畫（A\* 因成本上限放棄時也回報），再不行走共線救援／forced。
   寧可留下局部的琥珀色殘留衝突，也不畫繞整張圖的環繞線（香港旋轉 30° 案，
   2026-07-19：原本三條鏈各有一條繞行比 10–28× 的全圖環繞；改後 ≤2.3×，rect 鏈
   改以 2 條 forced 呈現走廊內的真 Jordan 封閉）。附帶修正：新加坡 orig rect 一條
   4.5× 的隱藏繞行降為 1.2×、零代價。
3. **Rip-up & reroute**（PCB 佈線手法）：被封死（Jordan——端點被其他線圍成的面
   關住，數學上無不交叉路徑）時，拆掉最常撞到的牆線（單條→兩條成對），先畫被關
   的段、再讓牆線改繞；全部乾淨才提交。
4. **Restart-with-priority**（negotiation）：仍失敗的段在下一輪**全圖重排**時最先
   route（牆還沒築起來），原牆線改繞它們；至多 6 輪。
5. **窄縫救援**：最後仍無路的段以 60% 貼線距離再 A* 一次（交叉依然絕禁）。
6. **共線救援（使用者規則：萬不得已可以共線，在真的找不到更佳解時；但新的交叉
   絕對不可以）**：窄縫都救不回的段，**放寬「共線／平行貼線」約束**再試一次——先
   bounded-bend 候選（`conflictCount(..., crossOnly=true)`，依折數），再 A*
   （`routeLattice(..., allowColinear=true)`，`blockerOf` 放寬 `legsHug`）。畫得出＝
   與他線**共線但不交叉**（勝過殘留交叉）。**`legsCross`（交叉）與 `pointOnLeg`
   （壓點）照樣絕禁**——只放寬平行貼線，不放寬穿越。計入 `stats.colinear`（工具列
   「共線」）。共線是可接受的次佳解，優先於 forced。

以上全部失敗才留下 `stats.forced`（工具列「殘留衝突」，琥珀底光＋hover 說明）——這些是
連共線都無解的**深度 Jordan 封閉**（唯一出路必須穿越圍牆線＝交叉，rip-up 也解不了）。
實測全 225 城：共線救回 **23** 條（原會 forced），剩 forced 為 rip-up 都解不了的極限
（NYC/東京 JR/巴黎等密網）。`stats.fallback`（非法角度兜底）為 0。

## 衝突判定（否決候選，「路線之間不可重疊」）

1. **共線重疊／平行貼線**：兩腿同族（H/V/D+/D−）且**垂直間距 < minGap
   （預設 0.35×格單位）**、投影延伸重疊超過一點 → 衝突（幾乎同高/同寬貼線也擋）。
   **總量上限（硬規則）**：同一段與已畫路網的累積重疊不得超過該段繪製弧長的 **25%**；
   超過即淘汰候選（不是「多一個衝突」）。共線救援也不可越過此上限。
2. **線段相交**：不同族的兩腿有任何接觸都算衝突，**唯一例外是端點對端點**
   （共用節點處交會）。幾何交叉在骨架階段已成黃點節點，節點以外不准相交。
   **同一對線最多交叉一次（使用者規則）**：候選與**同一條**已畫線出現第二個交點
   （交點座標去重，跨在對方轉角上不重複計）→ 直接淘汰（`Infinity`），不是「多一個
   衝突」；lenient 兜底時每多一個交點 +50。8/16 方向若真的必須交叉，也只能交叉一次。
3. **壓過節點**：腿的內部（端點除外）掃過任何其他彩色節點（端點落他線內部同理）→
   **直接淘汰**；這會把本不相連的點與線畫成相連，屬拓撲錯誤，不能用 forced／共線救援接受。
4. **節點環狀順序不可變（rotation system，使用者鐵律）**：每個節點接的線的 **360°
   環狀順序**必須與輸入（縮減網格）的弦方向參考序一致。H/V/45° 吸附可以轉「出線
   角度」，但同節點上任兩條線不得因此互換位置（順序一變＝拓撲改變）。實作
   `orderViolation(seg, pts)`：節點 ≥3 條已畫線時，依參考弦角排序後的「實際出線角」
   序列必須環狀遞增（繞一圈至多一次下降）；出線角完全相同（貼線雙生）視為平手。
   所有採納點都過這關——`pickBest`（lenient 兜底改記 +200 罰分）、A* 結果、
   rip-up／rerouteAround、joint 重算（帶對方擬定形狀 override）、共線救援、窄縫
   救援與四個收尾 pass。
5. 段的處理順序：**長走廊優先**（替代路徑最少），同長依序，結果確定性。

**硬違規的最後兜底（lenient rescan）**：若所有候選都超過 25% 共線上限或壓過外部節點，
不隱藏該段（隱形線會讓白點浮空、更糟），改以放寬計分重掃一次——壓站記 +100、超限重疊
記 +50——選「最不壞」的候選畫出並標 `forced`（琥珀殘留衝突）。絕不悄悄畫成合法樣。

**候選選擇序（`pickBest`，pass 1 與衝突消掃共用同一函式；Rule 0 直線除外）**：
①**衝突數最少** → ②**近距貼線最短**（`nearOverlapLen`，間距 <0.55 格的平行併走——
使用者裁決：視覺上的貼線也是「重疊」，且**排在 45/22.5 取捨之前**，「右邊可以用 22.5
解決重疊卻沒有」＝貼著別條線跑的 45 不得贏過完全不貼線的 22.5）→ ③**45 優於 22.5°**
（skew）→ ④**折數最少**（盡量直線）→ ⑤**硬共線重疊最短**（4 方向為了不重疊，
25%/75% 可在這裡蓋過 50%）→ ⑥**分點 50% → 25% → 75%** → ⑦軟共線最短（一格內近平
行）→ ⑧**路徑總長最短**（使用者規則 2026-07-20：線要越短越好——放在**最弱一階**：
其他準則全平手時才挑較短的繞行；曾試放在折數之後，東京 JR ilp 8 方向 forced 5→8、
翻動既有裁決，改到最弱一階即回穩）。重疊都相同時 50% 是預設
轉折位置。`rerouteAround` 的乾淨候選同樣先比近距貼線再比 skew；`l45Pass` 轉 45 不得
新添近距貼線。**共線救援**（forced 段最後手段）在所有「只剩共線、無交叉」的候選裡選
**重疊最短**的那個。
> 夾在兩排既有線正中間的段（如兩排相距一格的方框對角線）左右都貼半格，任何候選
> （含 22.5——軸腿會整段壓上其中一排）都不會更好，維持置中的 45–H–45 是正解。

**畫完仍有重疊 → 移分點（`overlapReducePass`，收尾第一步）**：任何畫出後仍有硬共線
重疊的段（含 forced），在**同折數**的兄弟候選（50%↔25%↔75% 分點、VH↔HV 順序）間
再試一次——衝突數不得增加、硬共線必須嚴格變短才換。三個方向級同一套判斷，實務上
主要在 4 方向發揮（使用者規則：4 方向畫出來有重疊，就試著移到 25%/75% 減少重疊）。

**規則 0：直線鐵律（使用者規則）**：端點在目前方向級下可直接以 H／V／45° 相連的段，
**且對已畫路網零衝突**時，一定直接畫成直線並鎖定。有重疊／交叉的「看似直線」不得鎖定——
必須改用 50%／25%／75% 或其他候選繞開；超過 25% 共線或壓站則淘汰。22.5°／67.5° 永遠
不是可鎖定走廊。兩條直線互撞時：先畫且乾淨者保留直線，後者改折，絕不兩條都鎖成重疊。

## 軟目標（衝突消解後微調）

**同名路線盡量不轉折（使用者規則）**：同一條路線穿過節點時盡量直線續接。
排在重疊／折數選擇之後（不會為此增加共線或折數），落實在三處：

1. **順接軟調整**：在「同轉折數」的無衝突候選間互換（腿的族別可換——L 形 ↔
   斜＋直都行），分數＝與共站同路線折線的續接角，**轉折角越大越好（使用者規則：
   先 180 再 135 再 90 再 45）**——直線續接(180°) +1、135° +0.707、直角 0、
   銳角 45° −0.707，計入 `stats.swapped`。
   段內轉折角同理：同折數候選序中，內角 135°（45° 接直線，兩個方向）都排在
   內角 90°（純 L 形）之前。
2. **單折 45° 端的擺位**：初始固定序（45° 靠 a 端優先），軟調整以續接分數改判——
   45° 靠在能與同路線鄰段斜線同方向連續的那端。
3. **L→45 是最後一步、能改就改（使用者規則）**：水平接垂直（單折 L）只要改成
   「45°＋直線」不與任何線重疊**就必須改**；兩個變體間以「同路線同方向 45° 鄰段」
   的那端優先，其次整體續接分數。它排在收尾迭代的**最末**，結果不再被其他 pass 改動。

收尾迭代：〔**A\* 樓梯轉 45（destairPass，最先）** → **22.5 升回 45（deskewPass）**
→ 壓短共線 → 順接軟調整 → 降折到定點 → **L→45（最後）**〕跑兩輪，
每步挪動走廊後回頭重撿。

## 黑點

黑點不是折線頂點：第 j 個（共 k 個）放在選定折線**弧長** j/(k+1) 處
（方向與 segment a→b 一致）。共用黑點（環線/折返路線同站出現在兩段）以後處理者為準。

## 實作契約

- `buildRwdMap(segs, pos, opts)`：`segs` 來自 [[route-hillclimb]] 的
  `buildHcGraph(skeleton, cellOf).segs`（含 `interior` 黑點 id 與父 edge 參照）、
  `pos` 為**像素座標** `Map<id,[x,y]>`（縮減網格格心）、`opts.unit` 為繞行偏移單位
  （像素，呼叫端傳 ~一格）、`opts.minGap` 為平行貼線門檻（預設 0.35×unit）。
  `unit` 內部取絕對值兜底——面板收合／拖曳的暫態尺寸會算出**負的 cell 寬**，負
  minGap 會讓 legsHug 永遠 false、貼線防護整個失效（畫出互相重疊又不標衝突的線）；
  D3Tab 端也在 host 尺寸 < 80px 時直接跳過 render（等 ResizeObserver 量到真尺寸）。
  dev 診斷：`window.__rwdAudit = true` 時 `stats._audit` 列出畫完後仍有殘餘衝突或
  標了 forced 的線（找「有衝突卻沒標」的洩漏）；`window.__rwdDebug` 為最新
  `cachedRWD`（D3Tab dev 模式掛上）。回傳
  `{ lines: [{seg, pts, bends, fallback, forced}], posAfter: Map<id,[x,y]>,
  stats: {straight, single, double, fallback, forced, colinear, swapped, segs} }`。
- 純函式、無亂數；同輸入同輸出。
- 繪製沿用 D3Tab 的路線著色（單線實色、共線交錯虛線）與節點分類色。

## 權重驅動版面簡化（論文 §九，右側「權重」tab，v1 已實作）

版面簡化**不改拓撲**：weight（流量／重要性）→ 非均勻欄寬列高 → 在**新像素座標重跑
buildRwdMap**（八方向約束以版面 pixel 為準）。UI：StylePanel 的「權重」tab（`viewKind==='rwd'`
時、物件 tab 之後），控制 D3Tab 的 `rwdWeightMode`／`regenRwdWeights`。實作：
`src/stores/rwdWeight.js`（純函式）。

- **weight 粒度＝相鄰兩站（link），不是 cut-to-cut 段的兩端點**（使用者規則：weight
  是 2 個站點就要有）。一個 cut-to-cut 段展開成站鏈 `[a, ...interior 黑點, b]`，鏈上
  每一對相鄰站是一個無向 link（`segLinks`／`linkKey`）。
- **weight 產生**：`randomWeights(segs)`——每個 **link** 抽一次 1–9，**反等比**
  （機率 ∝ 1/2ᵏ，小數字常見）→ 少數主走廊、多數次要邊。「全部隨機」整表重抽。
- **數字顯示**：只要有 weight 就一定顯示（不限 weight 模式），標在每個 link 兩站連線
  中點（interior 黑點位置：RWD 用 `posAfter`、縮減網格用 `placeBlacks`）。
- **每 5 秒自動重抽**（`setRwdAutoShuffle`）：開啟後 `setInterval` 每 5 秒
  `regenRwdWeights` 整表重抽 → 動畫過渡到新版面（離開 tab／unmount 清 timer）。
- **動畫過渡（§8.3，`animateToWeights`）**：weight 改變不瞬跳，而是 `requestAnimationFrame`
  ~700ms 內**內插起點 axes → 目標 axes 的格線位置**（`uniformAxes`／`lerpAxes`），每幀用
  內插後的 cellPx 重算 pxPos 並跑 `buildRwdMap(..., {fast:true})`。**內插的是欄／列格線
  比例，不是折線頂點**——否則中間幀違反八方向。`opts.fast`（rwdMap.js）中間幀只跑一次
  `routeAll`（含 4 輪衝突消掃），略過 6 輪重排／rip-up／窄縫救援／pass 2 軟調整，換每幀
  夠快；**最後一幀（t=1）走完整品質**（無 fast，settle 到無交叉版面）。動畫幀不進 RWD
  快取（sizeKey 含 `a{t}`）。權重改變重繪（含動畫幀與收尾完整 bake、以及權重網格模式
  下重建）**不蓋**「RWD 路網畫線中」overlay——版面／方形首次完整 bake 才用 busy 提示。
- **非均勻欄列**（`weightedAxes`）：欄寬列高由 link weight **取 max 不取 sum**（這一欄
  多重要看最忙那條）；方向相容——算 X 欄看 H/45° link、Y 列看 V/45° link。interior 黑點
  無格座標 → 依鏈上索引在 a→b 之間**線性內插**取該 link 所跨的欄／列。權重 0 的欄／列
  不消失（保留 `minFrac`≈0.25× 均勻寬，其餘依正權重分配）；外框固定。
- **模式**：`uniform`（均勻，預設）／`weight`（顯示 weight 比例）。切換或重抽都重跑
  buildRwdMap；快取鍵含模式＋重抽序號。
- **自動隱藏白點（§6，`opts.hideStops`／`opts.minStopPx` 預設 5px，使用者裁決＝站距決定
  全域 cutoff、全圖一致）**：兩階段——① **逐段**算「讓該段均分後站距 ≥ minStopPx 所需的
  最小 weight 差門檻」（`total/(k+1) ≥ minStopPx` 的寬鬆段需求 = −1），**全域 cutoff T ＝
  各段需求取 max**；② **全圖**任何白點只要左右 link 的 weight 差 |左−右| ≤ T 就隱藏。這樣
  「刪到 ≤ T」與畫面**完全一致**——寬鬆段的低差白點也一起消失（曾試過純逐段：寬鬆段殘留
  低差白點、與讀數矛盾，使用者否決）。**彩色錨點（a/b，紅／藍／黃）絕不隱藏**；保留的白點
  沿弧長**重新均分**；被藏的也放回原弧長位置。回傳 `hidden`＋`hiddenMaxT = T`。D3Tab 不畫
  這些白點、weight 數字合併跨過的可見路段標所跨原始 link 的 **max**。UI：權重 tab 的「自動
  隱藏白點」勾選＋「最小站距 pt」數字＋讀數「目前最小站距 高／寬 pt」「已隱藏 N 站 · 目前刪到
  weight 差 ≤ T」＋**被隱藏站名清單**（`stopStat.hiddenNames`，可捲動）。動畫每幀也套。

## LLM 調整（「AI 改網格長寬」，第三個視圖 tab，v1 已實作）

第二種「權重＝大小」比例來源（第一種是上面的流量 weight）：使用者說一句話
（「把市中心那幾欄拉開」），LLM 推理**每個 X 欄／Y 列區間**的顯示權重——不改
拓撲、不搬格座標，只改顯示上的欄寬列高。細節（執行流程、權重規則、觸發端點）
是 [[route-llm-grid]] 的權威範圍；這裡只記與本 skill 共用的末端變形：

- 結果檔 `data/metro/rwd-llmgrid/<city>.<variant>.<compact>.json`（colW/rowW/model/
  note/userPrompt＋fingerprint）；離線由 `scripts/llmGrid.mjs` export/apply 產生，
  網頁只載入（GH Pages 亦然）；dev 下「開始 LLM 調整」按鈕 POST `/llm-grid/run`
  spawn headless Claude Code（vite.config.js `llmGridTrigger`，與 `/llm-align`
  共用 `claudeSkillTrigger`）。
- **末端變形與流量權重完全同一套**：區間權重 → `intervalAxes`（rwdWeight.js，
  與 weightedAxes 共用 `axisEdges` 正規化：外框固定、minFrac 保底）→ 新像素座標
  重跑 buildRwdMap（非均勻 → 不傳 lattice）→ 新結果到達時動畫過渡（內插格線、
  fast 幀，同 §8.3，最後一幀完整品質）。
- UI：D3Tab `mode==='rwd-llm'`（tab badge 顯示模型名）、StylePanel「LLM調整」tab
  （一句話輸入框＋執行鈕＋結果紀錄）、畫布 overlay（執行中串流／尚未產生時的
  輸入框）。fingerprint（verts/segs/cols/rows）不符時顯示「與目前資料不符」。
