# 端點移動 (route-endpoint-move)

movewise 三步鏈（端點移動 → [[route-line-compact]] → [[route-grid-merge]]）的
**第 1 步**。輸入 = 該鏈（直角爬山/軸對齊/整數規劃/LLM 對齊）的結果佈局。
實作：`src/stores/hillClimb.js` 的 `buildEndpointStraighten(skeleton, cells,
cols, rows, {limit, skip})`（單掃描 pass；函式名保留歷史名）。

## 規則

- **每個非白點都可移動**（所有彩色頂點——端點/轉乘/分歧/黃色交叉；白/黑
  直通站不是頂點）。
- **候選＝四個 1 格單位移動**（使用者規則：每次移動不可超過 1 格——遠處的
  對齊靠 movewise 逐移動壓縮把距離拉近後慢慢完成）。
- **採納條件三擇一**：
  1. 自身入射段的 **H/V 淨增 > 0**（全圖 H/V 單調遞增；非入射段不受影響）。
  2. **H/V 數不變、但直線變長且斜線變短**（兩者都嚴格改善——典型：直線接
     斜線的轉折點沿直線方向推一格，一步步把斜線吃掉）。
  3. **v 是藍點（單一入射段）且線變短**（把端點往鄰居收）。
- **bendsPaid**：既有 H/V 段只有在「同一步有同路線的段被拉直」時才准折彎
  ——犧牲 X 線的直段去拉直 Y 線即使淨增也不行（sharesRoute 判定）。
- [[route-span-cap]]：入射段兩顏色點移動後不得橫跨超過上限。
- 每步走 makeMover **同一套硬規則**（不新增交叉、不壓到別的線/點、象限與
  邊環繞序不變＝拓撲不變）。
- tie-break：淨增大者 → 順著同 route 在鄰居另一側已是 H/V 的段接直
  （contScore）→ 長度改善量大者 → 位移小者 → 座標決定序。

## 收斂

每步讓 **(−H/V 數, 斜線長, 總長) 字典序嚴格下降**：條件 1 降第一項、
條件 2 降第二項、條件 3 不增第二項且降第三項——單調有界 → 終止
（movewise 壓縮會改變長度上界，MOVEWISE_CAP=5000 保險）。

## 歷史裁決（2026-07）

- 一次只能移 1 格（原本可吸到任意遠鄰居的欄/列）。
- 「直線變長、斜線變短就要移」與「藍點線變短可移」是後加的採納條件；
  藍點規則曾與舊「中位集中」拉鋸不收斂（33 城打上限），已隨網格合併
  重定義消失。
- 改名：端點拉直 → 端點移動。
