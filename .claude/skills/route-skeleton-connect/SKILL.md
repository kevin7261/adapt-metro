# 路線圖轉換骨架2（connect 骨架）(route-skeleton-connect)

把「選擇路線圖」的地理路網，收成示意圖佈局用的 **connect 骨架**：共線併成一條邊、真交叉補點、
紅／藍錨點保留、中段站變黑；並在骨架邊上標出**粉紅（轉折）**與**灰（過長黑點段分隔）**。
此 skill 是這個轉換的**唯一權威依據**，實作要與此處一致。

> **不拉直**：保留地理折線形狀，只做**拓撲收縮與標記**（與早期「直線骨架」不同——那個會拉直）。
> 純幾何/圖論轉換、**零網路、零 token**：只讀已載入的 GeoJSON（[[metro-osm-fetch]] 產出），
> 不改原始資料，輸出節點分類／邊分類／標記點供繪製；可即時開關（骨架化 ⇄ 還原）。
> 實作：`src/stores/skeleton.js` 的 `buildConnectSkeleton(geojson)`；UI 在 `D3Tab.vue` 的
> 「骨架化」按鈕（Map Adjust 群組的 D3.js 視圖）。

## 資料前提

本資料的線幾何＝「共站合併後依站序連線」（[[metro-osm-fetch]]）。圖論的「節點＝車站」、
「邊＝route 站序中相鄰的兩站」，共線（重疊路段）已由資料端去重成 `route_count≥2` 的路段。
每條 route 的站序讀 `routes[].stations`。

> **⚠ 折點不一定是車站**：多數折點是車站，但**跳站特快段例外**——特快沿本地線的真實
> 走廊彎行卻不停中間站，那段線 feature 的幾何會有一串**非車站折點**（Sydney North Shore &
> Western 快車 Strathfield→Redfern 段有 11 個折點，只有頭尾兩個是停靠站）。這牽動「線怎麼畫」：
> **建圖／分類只看 `routes[].stations`（停靠站），但畫線一定用 feature 的原始幾何 `path(f)`，
> 不可用「相鄰停靠站直線相連」**——否則特快跳站段會被拉成一條橫越空白的長直線（骨架冒出
> 原始沒有的線）。見下方「線一律照原始畫法」。

## 節點分類（依圖的 degree，不是 station_role）

以「route 站序相鄰兩站」建無向圖；`degree` = 相異鄰站數；每條邊記「經過的 route 集合」。

| 顏色 | 條件 |
|--|--|
| 紅 | 分歧／轉乘：`degree ≥ 3`，**或** `degree = 2` 但兩側邊的 route 集合不同（route 換線點） |
| 藍 | 真端點：`degree ≤ 1` |
| 黑 | 直通中段站：`degree = 2` 且兩側 route 集合相同 |

為什麼這樣分（**不用 station_role**）：
- 共軌段上「多線一起過、前後站相同」的站，鄰居仍只有兩個 → **黑**，不會整條共軌冒一串紅。
- 一條線被切成多段、段接點若 `degree=2` 且同 route → **黑**，串成連續骨頭，不被當藍端點打斷。

紅／藍是**強制骨架節點**；收縮時**絕不穿過**。

## 演算法

```
① 建圖、分節點（紅/藍/黑）
② 找真交叉 → 黃點
③ 收縮 degree-2 黑點鏈成「邊」（保留原折線形狀 + 經過的 route）
④ 邊分類（共線紅底 / 環線綠 / 頭尾共點藍）
⑤ 切紫點（藍 1/2、綠 1/3·2/3；共線不切）
⑥ 標粉紅（轉折）與灰（過長黑點段分隔）
```

### ② 找真交叉，補黃點（已實作，會切邊）
對每兩條 route 檢查所有線段對是否在**內部**相交（參數 t、u 皆嚴格落在 (0,1)，
故共站端點不算）——交點不是任何車站頂點 → **黃點**。**含同一條 route 的自相交**
（路線折回壓過自己）：以 `b = a` 的一趟做，同 route 內 `j` 從 `i+2` 起（不與自己或
共頂點的相鄰段比、每對只比一次）。以四捨五入座標去重、先 bbox 快速排除。
**黃點是真正的圖節點、會切斷邊**：造一個合成 id（`x0`、`x1`…）、座標＝交點，**插進 route
的站序**（跨線→插進兩條 route；自相交→同一條 route 的兩個段位各插一次，該黃點成 degree 4）。
依段內參數 t 排序、同段可多點。之後建圖時它自然成 degree≥3 的節點，收縮時邊在此切開；
分類時**強制覆寫為 `yellow`**（不套 degree 規則）。合成節點無 `station_name`，座標由回傳的
`crossings` 提供。繪製、格網化都把它當節點（前端把 `crossings` 併進 `posById`）。
（metro 多數交叉即轉乘站；自相交少見——台北 0、NYC 原始幾何 15 處。
**注意：只偵測「原始地理折線」上的交叉**；示意格網化/爬山法拉直後才產生的交叉不在此列。）

### ②'' 方向性重複路線（directional duplicates，已實作）
跳站特快（含「同一條線去/回程停站數不同」的短方向）沿真實走廊彎行卻不停中段站，
其線 feature 幾何有**非車站折點**（見「資料前提」）。**曾誤以拓撲修正處理**（把跳站
相鄰對「不建邊」，或把特快改走本地站序）——都動了拓撲、且非使用者要的。**正解是純
繪製**：骨架視圖的線一律照原始 feature 幾何 `path(f)` 畫，跳站段自然沿走廊彎、不會被
拉直成長直 chord。拓撲（建圖/分類）完全不動。

### 線的畫法（骨架 vs 原始，D3Tab）——**關鍵**
使用者的心智模型：**「原始怎麼畫，骨架就怎麼畫，只是共線變一條線」**。「一條線」＝共線
本來就是**一筆 feature**（`route_count≥2`）→ 一條折線；**顏色照舊交錯彩色**（不併成單色！
使用者明確要「顏色一樣」）。照做：
- **原始** ＝ Metro Maps 一模一樣：線用 feature 幾何 `path(f)`；單線實色、共線段
  （≥2 相異色）交錯彩色虛線。**無襯底、無節點分類**。
**核心原則：所有視圖的「線」一律畫同一批 line feature、同樣的顏色（單色實線／共線交錯
彩色虛線），差別只在「每個 feature 頂點的座標放哪裡」。** 骨架標記（節點分類色、邊分類
襯底）另外疊上。**絕不用「相鄰停靠站直線相連（edgeD 拓撲邊）」畫線**——那會（a）把跳站
特快段拉成橫越空白的長直線、（b）讓共線的交錯樣式與合併方式跟原始不同（使用者多次回報
「格網化後跟前不一樣／共線被分出來」）。

- **節點在地理座標的視圖**（純骨架 `mode==='skeleton'`、格網化前 `grid-*-pre`）＝ 用
  `path(f)`（d3 geoPath）畫 feature，**和原始一模一樣**；外加 ① 節點分類色、② 邊分類襯底
  （見 ④，沿 `e.geom` 畫、僅此類視圖有）。
- **格網化後 `grid-*-post` / HC / 縮減網格** ＝ **畫同一批 feature、同樣顏色**，只是把每個
  feature 頂點的**車站**改對應到它的**格子/HC 座標**（`movedFeatureLines`／`featMovedD`：
  coord→station id→`movedOf`）。所以「後 ＝ 前的同一張圖、只是座標搬到格子上」——線數、
  顏色、共線交錯、哪些線併在一起全部一致（雪梨 前後都 79 線 10 色）。非車站頂點（真實
  軌道折點）在格子上無對應→略過、格點間直連。**黃點（②交叉）不是 feature 頂點**，是落在
  某條「相鄰車站段」上的合成節點——畫線時要在該段幾何檢測（共線且 0<t<1）把黃點的
  **格子座標插進去**，否則格網化後線會跳過黃點格、黃點浮空、兩線不在該處相交（使用者回報
  「黃點位置錯、線該斷沒斷」）。NYC 96 黃點中 ~88 修好；少數落在**跳站特快**段（黃點算在
  停靠站直線 A–D 上、但 feature 走 A–B–C–D）抓不到，屬已知邊界。**幾何**跟原始不同是**應該的**（示意圖化），
  但**線與色必須跟前一致**。無邊分類襯底（座標已搬、`e.geom` 失效）。
- **⚠ 兩份繪圖程式**：主視圖在 `D3Tab.vue`；**全球視圖畫廊縮圖**是另一份獨立實作
  `src/stores/viewGeometry.js`（`computeCityViews`/`computeCityHcViews`，即時在瀏覽器現畫、
  無快取檔）。**改繪製規則（線幾何來源、襯底、顏色）必須兩邊一起改**，否則畫廊縮圖會停在
  舊行為。畫廊同樣：地理座標視圖用 `featureLines`(=path(f))＋`geomHl`(=沿 e.geom 襯底)，
  格網化後用 `edgeD`。

**邊帶真實幾何 `edge.geom`**：`buildConnectSkeleton` 為每條邊算出**真實折線座標串**
（`[[lng,lat],…]`）——把 feature polyline 依「該 route 的**停靠站**」切段（不是依每個站
頂點！特快 feature 穿過它跳過的站，那些站頂點不是它的停靠站，故 Strathfield→Redfern 這條
邊擁有整段穿過中段站的 9 折點曲線）。襯底就沿 `edge.geom` 畫，貼著線彎、不會在跳站段
戳出直線。無 feature polyline 的段（如被黃色交叉點界定的腿）退化為兩端直線。

### ②' 骨架不加線（核心原則）
骨架是 **Metro Maps 網路的純拓撲收縮**——**絕不新增** Metro Maps 沒有的邊/線。
曾試過「express-over-local」：把跳站特快途經的本地停靠站（HK Airport Express 途經
欣澳/荔景…）用弦鄰近插進特快站序，想合併共軌走廊；但這會**捏造岔口與邊**（欣澳變
degree-3）、讓骨架的線**比 Metro Maps 多**——錯的，已移除。Metro Maps 畫成分開的
跳站特快/本地線（因站序不同），骨架就分開；只有資料裡本來 route_count≥2 的共線段才併。

pass/stop 只作**顯示**、不動拓撲：**`computePassThrough(geojson)`**（skeleton.js 匯出，
純函式，弦鄰近測試）回傳 `{ stopByStation, passByStation, seqByRoute }`——每站停靠/
通過的路線、每條路線的合併站序（stop/pass 標記）。物件 tab（StylePanel）用它列出
車站的停靠/行經路線、路線站序中標 pass 站。**弦近似本質不精確**（欣澳抓得到，
Nam Cheong/Olympic 因走廊彎曲略超容差），要精準需資料層 stop/pass（見 TODO）。

### ③ 收縮成邊
從每個紅/藍節點沿每個鄰邊，一路穿過 `degree=2` 黑點直到下一個節點——這串站＝一條**邊**。
邊保留**原折線（地理形狀不變）**與其**經過的 route 集合**；中段黑點仍在原位置畫，不消失。
純環（整條 route 皆黑、無紅/藍）自成一條首尾同節點的**自環邊**。

### ④ 邊分類（線底下的 highlight 襯底）
**線色不動**（單線＝route 原色、重疊＝各 route 交錯彩色虛線 `dasharray [0,i·D,D,(n−1−i)·D]`、
n 依 MAX_OVERLAP=6 夾住）——與 metro map 完全相同。邊分類畫成**線底下的 highlight 襯底**：
較寬（`strokeWidth+11`）、半透明（`opacity 0.55`）的底色線，**沿 `edge.geom` 真實折線**畫
（貼著線彎、跳站段不戳直線）。純骨架地理視圖才畫；格網化/HC 視圖不畫（`highlightData=[]`）。

| 類別 | 條件 | 襯底色 |
|--|--|--|
| 共線合併 coline | 邊上 route 有 **≥2 種相異色**（非只看 route 數）；**不切紫點** | **紅** `#e11d48` |
| 環線 loop | 邊首尾同節點（自環）；**優先於**頭尾共點 | **綠** `#16a34a` |
| 頭尾共點 parallel | 同一對節點 A–B 間有 ≥2 條不同幾何邊（平行多重邊），且非環線 | **藍** `#2563eb` |
| 一般 plain | 其餘 | 無襯底 |

> 曾一度整個拿掉襯底（`highlightData=[]`），因為舊版用「車站點直線相連」畫襯底，在跳站段
> 戳出橫越的粗色帶、共軌密集城市（雪梨）被蓋滿。**根因是幾何來源錯**（該用 `edge.geom`
> 折線、非車站點直線）——改對幾何來源後襯底恢復，使用者要的「邊分類襯底」回來了。

> **紫色襯底不是類別**：襯底半透明（0.55），當**紅（共線）與藍（頭尾共點）兩條不同邊
> 的 `edge.geom` 在同一走廊重疊**（如特快 parallel 藍邊壓在一段 coline 紅邊上；雪梨有 5 對
> 這種重疊），兩層半透明底色**混色成紫**。這是渲染疊色、非第四種分類。若要消紫：讓 coline
> 優先（重疊處不畫 parallel），或降不透明度/加寬度差，屬顯示層取捨。

> **共線判定用「相異色數」而非「route 數」**（與 metro map 的 `_nc` 一致）：同一條線被拆成
> 多個同色 route_id（如新加坡環狀線 CCL 的主環 route_id 與 HarbourFront↔Dhoby Ghaut 段
> route_id）在同一邊上，**不算共線**、畫成單線。否則會被當共線畫成並排平行線，在
> HarbourFront 那種「並排段接單線閉合段」的交界處看起來斷開。

### ⑤ 紫點切斷
| 邊類型 | 切斷位置 |
|--|--|
| 共線合併 | 不切 |
| 頭尾共點（藍） | 弧長約 **1/2** 處切 1 個 |
| 環線（綠） | 弧長約 **1/3、2/3** 處切 2 個 |
取點：取最接近該弧長比例的內部黑點升為**紫**節點。

### ⑥ 粉紅（轉折）與灰（過長段分隔）
**粉紅＝代表性轉折點**，從骨架邊上的中段黑點挑出，只看「這條邊夠不夠彎」＋「哪幾個彎夠代表性」。
逐邊兩道關卡（與路線大小無關，只看相對彎曲）：

1. **曲折度關卡**：曲折度 = **弧長 ÷ 兩端直線距離（弦長）**。≈1 幾乎直線、越大越彎；頭尾幾乎
   重合（環）→ 弦長≈0 視為**極曲折**。**曲折度 ≤ 1.25（弧只比直線長約 25% 內）→ 這條邊不挑粉紅。**
2. **夠彎 → Douglas–Peucker 找代表性轉折**：容差是**比例不是公尺**——子段頭尾連線當基線，中間
   最遠點的 **垂距 ÷ 弦長 > 0.25（約 25%）** 才保留該點，再對左右兩半遞迴。
3. 保留的中段頂點**本身必須是黑點站**才標粉紅（不是站的幾何頂點不標）。
4. 同一黑點出現在多條共線邊 → 取較大的轉折比例（被任一邊保留即為粉紅）。

常數：`PINK_SINUOSITY = 1.25`、`PINK_DP_TOL = 0.25`（`src/stores/skeleton.js`）。
- **灰＝過長黑點連續段的分隔**（規則）：任兩相鄰「邊界點」之間，黑點最多 **4** 個。
  邊界＝邊的兩端真實節點 ＋ 內部粉紅（灰也成為之後的邊界）。
  一連續段有 **N** 個黑點時，改灰的個數 **G = ⌊N / 5⌋**（4+1=5；N=5→1、N=10→2…）；
  灰點在段內盡量均分（偏中間）。**優先序：已是粉紅的不改灰**。

## 骨架畫面上的點與邊

| 顏色 | 角色 |
|--|--|
| 紅 | 分歧／轉乘（degree≥3，或兩側 route 不同的 degree-2） |
| 藍 | 真端點（degree≤1） |
| 黃 | 幾何交叉新點 |
| 紫 | 頭尾共點／環線切斷點 |
| 黑 | 直通中段站 |
| 粉紅 | 曲折邊的代表性轉折黑點 |
| 灰 | 過長黑點段中間的分隔點 |
| 紅底邊 | 共線合併 |
| 藍邊 | 頭尾共點分歧 |
| 綠邊 | 環線 |

## 使用流程（UI）

1. 先在 Metro Maps 匯入城市（或匯入 GeoJSON）。
2. 在 **Map Adjust** 的 D3.js 視圖選定來源（或直接匯入檔案）。
3. 按 **「骨架化」**（再按一次還原）；可與「依建議旋轉」並用。

## 實作契約（改功能時遵守）

- `buildConnectSkeleton(geojson)` 為**純函式**：不改輸入；回傳
  `{ stationClass: Map<id, 'red'|'blue'|'black'|'purple'|'pink'|'gray'|'yellow'>,
     edges: [{ path:[id…], geom:[[lng,lat]…], cls:'coline'|'loop'|'parallel'|'plain', color, routeColors }],
     crossings: [{ id, coord:[lng,lat] }] }`。`stationClass`／`edges` 的 path 可含合成交叉 id
  （`xN`，見 ②）；`crossings` 給前端解析其座標。`edge.geom` ＝該邊的**真實折線座標串**（依
  route 停靠站切段、含跳站段穿過的中段站折點），供**地理視圖的邊分類襯底**沿線畫。座標一律
  取原座標（**不移動、不拉直**）。（線本身不用 edges 畫——見上「核心原則」，一律畫 line
  feature；edges 只供節點/邊分類與襯底。）
- degree／route 集合一律由 `routes[].stations` 建圖得出（不讀 `station_role`）。
- 已實作：①②③④⑤⑥（節點/邊分類、黃色幾何交叉、紫點、粉紅、灰）。②黃點是幾何標記，
  只在純骨架視圖畫、格網化視圖不畫（見 ②）。
