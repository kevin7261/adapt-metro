Adapt-Metro 系統架構

本頁以 C4 模型與 UML（元件圖、類別圖、狀態機、序列圖、ER 圖）完整描述 Adapt-Metro 的系統架構： Vue 3 單頁應用、Vite dev server 中介層、Node.js 資料管線、Claude Code LLM 整合， 以及「Metro Maps → Map Adjust → Straighten → RWD Maps」四階段示意圖佈局演算法管線。 圖中的識別字（檔名、函式、路由）皆與程式碼一一對應，可直接以此頁為 code tour 的入口。

FLOW四階段演算法管線（核心）

這是整個系統的主軸：把地理正確的路網 GeoJSON，逐階段收斂成嚴格 H/V/45° 的示意圖。 每一階段都是獨立的 dockview 視圖，可逐步檢視中間結果；每城市另有 orig（原始）與 rot（旋轉）兩個變體各跑一遍全管線。

後續的「直線演算法」九條鏈（§9）都以此結果為輸入做 H/V 最大化後處理；其中②直角爬山 （buildRectPolish）就是把方向準則從八方向 |sin 4θ| 換成直角 |sin 2θ| 再爬一次，經 iteratePost 迭代到不動點（上限 POST_ITER_CAP = 20）。

UMLmovewise 三步鏈循環（狀態機）

「循環」tab 的執行語意：三個演算法各自跑到不動點才輪替，一整輪完全靜止才停機。 每一個單一移動後立即壓縮網格（movewise），不再有獨立的縮減步驟。

RULE跨模組不變式與持久化

演算法不變式（違反即 bug）

  • 唯一移動關口：任何演算法要移動節點都必須經 hillClimb.js makeMover；四條硬規則（邊界／象限／遮蔽／邊環繞序）與 FROZEN 檢查都在這一個關口實施。
  • 成方凍結：⑨ 成方被「執行調整」採用後，成方頂點進入 SQUARE_GUARD（setFrozen）；下游所有鏈（②爬山、movewise、①〜⑧、RWD）可動方形頂點以外的點，但不得破壞四邊直線。
  • SPAN_CAP：端點移動與直線縮減的每一步，受影響段的兩個彩色點 Chebyshev 距離不得超過上限（預設 3；樣式 tab 可調＋重新計算套用）。
  • paper fidelity：直線化演算法照 data/thesis 論文 md 實作、不自創；只算彩色點，白/黑點事後平均放回。
  • 嚴格改善：論文鏈的逐批套用（finishBatches strict）只接受 scoreAlign 變好的批次——HV 優先、45° 次之。
  • LLM 結果不自動套用：eval／grid／align／shape 跑完只寫檔＋顯示，一律等使用者按「執行調整」；重跑清舊資料。
